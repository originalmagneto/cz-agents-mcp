# ISIR v1.1 — real `insolvency_count` for realestate — design

**Date:** 2026-05-31
**Status:** scoping (planning-only, not yet approved for build)
**Builds on:** realestate full-data (Phase 2b, done). Goal: make `get_district_aggregate.insolvency_count` real (currently always 0).

## Problem

`get_district_aggregate` counts `insolvency_count` = `RealEstateLead` rows where `sourceType='isir'` in an okres+window. We write none today, so it's always 0. The hard part is resolving an insolvency to an **okres** — ISIR's data model does not give okres directly.

## ISIR data reality (verified earlier)

- **ISIR PublicWS** (`pollEvents`) — append-only event firehose. Has spisová značka, typ/popis události, document URL. **No IČO, no address, no okres.**
- **ISIR CuzkWS** (`getIsirWsCuzkData`) — lookup by IČO / rodné číslo / name+DOB → debtor record **with address** (okres-derivable) + `druhStavKonkursu` + `urlDetailRizeni`. But it's a **lookup, not an enumerable feed** — you can't list "all insolvencies in okres X".
- The upstream cz-agents-webapp resolved okres via a document-parsing ("vision") pipeline we don't have.

So: there is no clean public way to enumerate *all* insolvency proceedings by okres. Two achievable tiers:

## Approach — two tiers

### Tier A (recommended) — insolvency-origin auctions from portál dražeb
The portál dražeb feed already carries a **`type`** field (verified: current "pripravované" snapshot is 20× `execution`; insolvency sales appear as a different `type`/auctioneer = insolvency administrator rather than `ex_number` exekútor). These are real-estate sales **with okres already resolved** (`location_district.district_name`) — the same pipeline we built.

Change: in `realestate-ingest`, classify each scraped auction by proceeding origin:
- exekútor (`auctioneer_office.ex_number` present / `type='execution'`) → `sourceType='portaldrazeb'` (auction_count) — unchanged.
- insolvency origin (`type` indicates insolvency / administrator, not exekútor) → `sourceType='isir'` → counts as `insolvency_count`.

Result: `insolvency_count` becomes a **real count of insolvency-driven RE auctions per okres**, reusing the existing scraper + DB + okres mapping. No new infra.
**Limitation (must be documented in the tool output / README):** this counts insolvency sales *that have reached auction on portál dražeb*, NOT all open insolvency proceedings. For realestate's distress-property purpose this is the relevant signal, but it is narrower than "all ISIR insolvencies in the okres".

**Effort:** small — ~1 day. Extend the parser's classification + a fixture case + tests; reuse everything else. Same deploy path (push → UI Deploy → ingest schedule).

### Tier B (full ISIR, larger) — ISIR event poll + okres resolution
Build a real ISIR pipeline: poll PublicWS for RE-relevant event types (dražební vyhláška nemovitosti, zpeněžení nemovitosti…), then for each proceeding resolve debtor→okres. Okres resolution options, all imperfect:
- (b1) parse the dražební-vyhláška document (PDF) for the cadastral area → okres — needs PDF text/OCR + a katastr→okres map. This is the webapp's "vision" approach.
- (b2) CuzkWS by debtor name (extracted from event) → address → okres — fragile name matching, rate-limited, lookup-only.

**Effort:** large — multi-day, fragile, needs PDF parsing + a katastrálne území→okres dataset (RÚIAN). High maintenance. Coverage still partial (only proceedings whose documents are parseable).

## ⚠️ Recon update (2026-05-31) — Tier A is NOT viable

Sampled all portál dražeb public feeds (`pripravovane`/`online`/`probihajici`.json): **100% `type=execution`**, every record has an exekútor (`auctioneer_office.ex_number`), including the `voluntary:true` ones. portál dražeb is the **exekútor** portal — it carries NO insolvency-administrator sales. So classifying by `type` would tag everything as execution → `insolvency_count` stays 0. **Tier A cannot produce insolvency data.** ISIR v1.1 therefore requires Tier B (real ISIR pipeline) or a different insolvency-sale source.

### Revised options
- **Tier B (real ISIR pipeline)** — poll ISIR PublicWS events, filter RE dražba event types ("dražební vyhláška", "zpeněžení", "nemovit"), fetch the dražební-vyhláška PDF from `dokument_url`, extract katastrální území, map k.ú.→okres via a RÚIAN/ČÚZK dataset (public). This gives the **property okres** (most correct). Multi-day, needs PDF text/OCR + a k.ú.→okres table; coverage = events whose docs parse.
- **Tier B-lite** — same event poll, but resolve okres from the **debtor address** via CuzkWS (query by debtor name extracted from the event). Less code (no PDF), but okres = debtor's residence, not the property's, and name-matching is fragile.
- **Alternative source research** — find a central insolvency-auction listing (insolvency správce sales) with structured okres, analogous to portál dražeb. Needs investigation; may not exist as clean JSON.
